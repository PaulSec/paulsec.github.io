Blackstone Smoker and Telegram Bot

Mar 3, 2026
10 min read

Building a Telegram BBQ Pitmaster with Live Grill Data and GPT-4o

In the first post, I covered how I reverse-engineered the Blackstone Connect API to stream temperature data into Grafana Cloud. That gave me dashboards, historical graphs, and alerting — a massive upgrade over the stock app.

But there was still friction. During a cook, I’d have to open Grafana on my phone, navigate to the dashboard, squint at the graph, and try to mentally extrapolate when my brisket would hit 203°F. I wanted something I could just ask. From the backyard. While holding a beer.

So I built a Telegram bot that sits on top of the same Prometheus data, runs linear regression on the fly, pulls live weather data, and pipes everything through GPT-4o to act as an AI pitmaster. This is how it works.

Why Telegram?

I considered a few options — a custom web app, a Slack bot, push notifications via Grafana — but Telegram won for a few practical reasons:

  • It’s already on my phone. No extra app to install.
  • The bot API is dead simple. python-telegram-bot handles all the boilerplate.
  • It supports rich text, inline keyboards, and markdown. Good enough for formatted temperature readouts.
  • It works from anywhere. My wife can check the cook status too without needing Grafana access.

The bot runs as a systemd service on a cheap VPS. I deploy with a single script. More on that later.

Architecture

The Telegram bot doesn’t talk to the Blackstone API directly. That’s still main.py’s job — it keeps polling the grill and pushing metrics to Grafana Cloud. The bot reads from Prometheus (Grafana Cloud’s query API) and layers intelligence on top.

Here’s how the pieces fit together:

The bot queries Prometheus for the last N minutes of data, runs its own analysis (linear regression, probe scoring, stall detection), builds a rich context string, and sends it all to GPT-4o along with the user’s question. The LLM doesn’t guess — it has real numbers to work with.

The Commands

The bot exposes a handful of slash commands. Each one is purpose-built for the kind of questions you actually ask during a cook:

Command What it does
/status Current temps for grill + all 4 probes, with trend arrow
/eta 203 “When will I hit 203°F?” — linear regression + LLM analysis
/remaining 165 Same as /eta, just different wording
/history 60 Last 60 minutes of temp data, mini ASCII bar chart
/stall Detects if you’re in the dreaded 150-170°F plateau
/probe Reliability scores for all 4 probes, auto-selects the best one
/setprobe 2 Override auto-selection, lock to Probe 2
/weather Current weather in Greer, SC + impact on your cook
/ask <anything> Free-form question with full data context

You can also just type a question without any command — the bot will pick it up and answer using live data.

Smart Probe Selection

This was one of those features I didn’t plan but ended up being essential. I have 4 probes but rarely use all of them. Sometimes one is dangling off the grill doing nothing, sometimes one has a flaky connection. I don’t want ETA predictions based on a dead sensor reading ambient temperature.

The bot scores each probe on 5 criteria:

  1. Temperature range — Is it between 32-500°F? (+30 points)
  2. Variance — Is the reading actually changing? A stuck sensor gets zero. (+20)
  3. Smoothness — How big are the jumps between consecutive readings? Erratic probes get penalized. (+20)
  4. Above ambient — A probe reading 72°F probably isn’t in meat. (+15)
  5. Data completeness — More data points = more reliable. (+15)

The probe with the highest score out of 100 gets used for all calculations. You can see the full breakdown with /probe:

If the auto-selection is wrong (it happens — maybe you just inserted a probe and it hasn’t warmed up yet), you can override it:

/setprobe 2     → Lock all calculations to Probe 2
/setprobe auto  → Back to automatic

The override is per-chat, so my wife and I can each track different probes from our own Telegram if needed.

Temperature Prediction

The ETA prediction is the feature I use the most. “When will this be done?” is the question during any long cook.

The approach is straightforward: take the last 30 minutes of data, run linear regression, and extrapolate to the target temperature.

# Normalize timestamps to minutes, fit a line
t_minutes = (timestamps - timestamps[0]) / 60.0
coeffs = np.polyfit(t_minutes, temps, 1)
slope = coeffs[0]  # °F per minute

But a naive linear fit isn’t great by itself. A few refinements make it actually useful:

Short-term vs. long-term trend. The bot computes both the overall slope (all data) and the recent slope (last 10 points). It uses the more conservative of the two. If the meat is slowing down, you want to know.

Stall detection. If the current temperature is between 150-170°F and the rate drops below 0.5°F/min, the bot flags it as a stall. This is huge — the stall can add hours to a cook if you don’t wrap.

Confidence scoring. The bot computes R² (goodness of fit). High R² with lots of data points = high confidence. Noisy data with few points = low confidence. This gets passed to GPT-4o so it can caveat its answer appropriately.

LLM layer on top. The raw regression numbers are a starting point. GPT-4o gets all of this data plus the current weather (wind, humidity, outside temperature) and combines it with BBQ domain knowledge. Wind at 20 mph? The LLM knows that means heat loss and will adjust its estimate.

Weather Integration

Outdoor cooking is weather-dependent. A 15 mph wind or 35°F ambient temperature will slow your cook noticeably. The bot pulls weather data from Open-Meteo (free, no API key needed) with OpenWeatherMap as a fallback.

The /weather command shows conditions and directly tells you the impact:

  • Cold temps → more pellet consumption, slower cook
  • High wind → heat loss, shield your grill
  • High humidity → better smoke absorption
  • Dry air → meat dries out faster, consider spritzing

This data also gets injected into every LLM prompt, so when you ask “when will my brisket be done?”, the AI factors in that it’s 38°F and windy outside.

The LLM Layer

Every command that involves analysis — /eta, /stall, /ask, and free-form messages — goes through GPT-4o (Azure OpenAI). But it’s not a generic chatbot. The system prompt gives it a very specific role and very specific data:

BBQ_SYSTEM_PROMPT = """You are an expert BBQ Pitmaster and intelligent cooking assistant.
You are monitoring a Blackstone pellet grill in real-time via temperature probes.
You are located in {location}.

Current weather conditions:
{weather_context}

Temperature data from the last 30 minutes ({active_probe} - meat, auto-selected):
{probe_data}

Current grill temperatures:
{current_temps}

Trend analysis (linear regression on {active_probe}):
- Current rate: {rate:.2f}°F/min
- Current temperature: {current_temp:.1f}°F
{stall_info}
"""

The prompt includes the full time-series data (30 minutes of readings with timestamps), all current probe values, weather, the linear regression rate, and whether a stall was detected. The LLM doesn’t hallucinate temperatures — it reads them.

The really nice thing about this setup is free-form queries. You can just type a message like:

“Should I wrap now or wait?”

And the bot will look at your current temperature, the rate of climb, whether you’re in a stall, and give you a real answer based on your cook, not generic advice.

How It’s Deployed

The bot runs 24/7 on a Linux VPS as a systemd service. Here’s the setup:

Dockerfile — minimal, just Python 3.11-slim with the bot script and config:

FROM python:3.11-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY telegram_bot.py .
COPY grafana_config.py .
CMD ["python", "telegram_bot.py"]

Systemd service — for bare-metal deploys. Auto-restarts on failure, runs as a dedicated blackstone user with hardened security (read-only filesystem, no privilege escalation, private tmp):

[Service]
Type=simple
User=blackstone
ExecStart=/opt/blackstone/venv/bin/python3 /opt/blackstone/telegram_bot.py
Restart=on-failure
NoNewPrivileges=true
ProtectSystem=strict
ProtectHome=true
PrivateTmp=true

One-command deploy — sudo bash deploy.sh handles everything: creates the system user, copies files, sets up a venv, installs deps, enables the service.

sudo bash deploy.sh
sudo nano /opt/blackstone/.env    # set your tokens
sudo systemctl start blackstone-telegram-bot

That’s it. Logs go to journald (journalctl -u blackstone-telegram-bot -f).

Things I Learned

Linear regression is surprisingly good for BBQ. Meat temperature rise is roughly linear within any 30-minute window (outside of the stall). Simple numpy.polyfit with degree 1 nails it. I tried polynomial fits — they overfit on noise and gave wild predictions.

Probe auto-selection saves you from yourself. I can’t count how many times I would have been tracking a dangling probe if the bot hadn’t flagged it. The scoring system is simple but catches the obvious failure modes: dead sensors, probes reading ambient, erratic connections.

Weather matters more than you’d think. A 20°F drop in ambient temperature or 15 mph winds can add 30-60 minutes to a long cook. Having the LLM factor this in automatically makes the estimates significantly more realistic.

Telegram’s bot API is a joy to work with. The python-telegram-bot library handles polling, command parsing, async execution — you just write handler functions. Went from zero to working bot in an afternoon.

The LLM needs data, not vibes. The whole reason this works is that GPT-4o gets actual numbers — 30 minutes of timestamped readings, computed rates, R² values, probe scores, weather data. When you give an LLM concrete data plus domain expertise in the prompt, the answers are genuinely useful. When you don’t, you get generic BBQ tips from the internet.

What’s Next

A few ideas I’m kicking around:

  • Cook session recording — save the full temperature curve for each cook with notes. Build a personal recipe database with actual data.
  • Predictive alerts — “Your brisket will hit 165°F in about 20 minutes, want me to remind you to wrap?”
  • Multi-grill support — for when I inevitably buy another smoker.
  • Photo analysis — snap a pic of the meat, send it to the bot, get a doneness assessment via vision API.

But honestly? The current setup does 90% of what I need. I pull up Telegram, ask “how’s the cook?”, and get a real answer with real data. That’s the whole point.

The full source code is on GitHub. The bot is telegram_bot.py — about 550 lines of Python including all the Prometheus queries, probe scoring, linear regression, weather integration, and LLM wiring. No framework beyond python-telegram-bot and LangChain for the OpenAI calls.
